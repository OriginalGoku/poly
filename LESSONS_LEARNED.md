# Lessons Learned

- Polymarket `/books` batch endpoint is `POST` with JSON body `[{"token_id": "..."}]`, not GET with query params. GET returns 400 "Invalid payload".
- Gamma API `/markets` endpoint ignores `tag`, `event_slug`, and `_q` params — always returns the same default results. Use `/events` with `tag_slug` instead; markets are embedded in the event response.
- CLOB `/trades` requires full API key auth (POLY_API_KEY + signature headers). The Data API at `data-api.polymarket.com/trades` returns public trade data without auth.
- Data API trade pagination params (`after`, `before`, `cursor`, etc.) don't shift the result window — may need timestamp windowing or CLOB API key for proper incremental fetching.
- `riot_esports_sample.json` fixture was saved as a placeholder (`{"note": "populate after successful API calls"}`) because no Riot API key was set — don't rely on it for tests.
- Data API (`data-api.polymarket.com`) has an undocumented rate limit much lower than the CLOB API — approximately 1 req/s sustained. Concurrent trade polling causes 429s even at moderate concurrency. Fix: add 1s delay between sequential requests + 429 retry with 2s backoff.
- `book_depth_usd` computed within ±5% of mid-price reads zero for wide-spread markets (e.g., 48c spread, mid=0.64). `inside_liquidity_usd` (best_bid * size + best_ask * size) is a more reliable metric for thin markets.
- Many Polymarket tokens return exactly 100 trades (API max) on first fetch — saturation monitoring is critical to quantify trade data loss.
- Polymarket WS Market channel initial subscription response is a JSON **array** of book snapshots (one per token), not a single object. Subsequent `book` events are individual full snapshots.
- WS `last_trade_price` event includes `transaction_hash`, `size`, `side`, `fee_rate_bps` — full trade metadata, contrary to initial assumption that it would only have price. WS-only architecture for trades is viable.
- Polymarket WS Sports channel has no `event_type` or `slug` field — messages are flat objects keyed by `gameId` (integer). Matching to config `match_id` requires a lookup or fuzzy team-name match.
- 88-token WS subscription on a single connection works without errors — but causes server-side disconnects every ~80s (23% downtime). The `websockets` library default `ping_interval=20` conflicts with custom text `"PING"` heartbeat, and high token count hits server bandwidth limits. Fix: use library ping frames only (`ping_interval=30, ping_timeout=10`), remove text PINGs entirely, shard to ≤25 tokens per connection (core vs prop markets), reset backoff counter only after receiving actual data (not just TCP connect).
- WS `book` events are full snapshots, not deltas — no need to maintain local order book state or apply incremental updates. Simplifies `OrderBookSnapshot.from_ws()` significantly.
- `from_ws()` class method factory pattern on dataclasses works well for dual REST/WS support — keeps parsing logic co-located with the model while allowing both data sources to produce the same type.
- WS write buffering via `asyncio.Queue` between the client and DB writer task decouples message dispatch from SQLite I/O, preventing slow writes from causing WS message backpressure.
- For dual-write validation (WS vs REST), use `UNIQUE(transaction_hash, token_id, source)` not `UNIQUE(transaction_hash, token_id)` — the latter causes whichever source writes first to win, making it impossible to compare capture rates. Both sources need to insert independently.
- REST trade polling fetches historical batches (up to 100 trades) on first call, inflating REST counts. When comparing WS vs REST capture rates, filter to only trades after WS connected (`MIN(server_ts_ms) WHERE source='ws'`) for a fair comparison.
- REST Data API trades have a market_id mismatch: returns event-wide trades, not filtered to config markets. In an NBA game only 3/3900 trades matched configured token IDs — drop REST trade polling after WS validation passes.
- `match_events` = 0 across all collected databases — game state clients (NBA CDN, OpenDota) are not producing events, blocking the overreaction hypothesis analysis.
- ~66% of order book snapshots have spread >10c. Focus analysis on ~22 liquid tokens per NBA game (moneyline, spread, O/U) rather than player props.
- WS `price_signals` provide sub-second BBO updates (0.13s avg interval) — the best data source for price overshoot detection, much higher resolution than order book snapshots.
- NBA game state produced 0 `match_events` because configs had `external_id` empty — the NBA CDN client needs a valid game ID. Fixed by adding `lookup_game_id()` that auto-resolves the game ID from the NBA scoreboard at collector startup.
- NHL configs all had `data_source: "none"` despite `__main__.py` having correct NHL dispatch logic — root cause was `discover_markets.py` not setting the field for older configs. Fix: patch configs, add warning log in `load_config()` for future detection.
- WS `price_change` events are order book deltas (resting order size at a price level), not fill notifications — `size: "0"` is common. Do not create Trade records from them.
- NHL API provides no absolute wall-clock timestamps — only game clock (`timeInPeriod`). Using a single `datetime.now()` per poll batch collapsed all events to one timestamp and introduced up to 10s poll-time bias. Fix: per-event `datetime.now()` + sortOrder sub-ms offsets for uniqueness, poll interval reduced to 5s.
- Polymarket Data API `asset_id` query param on `/trades` is **ignored** — returns event-wide trades regardless. Confirmed: REST batches of 100 trades contain 30-50 distinct token_ids from ~2,600 tokens across the entire event, not just the requested token. This is why REST shows <2% overlap with WS for configured tokens — REST captures mostly unrelated markets. WS captures 98.5-99.5% of configured-token trades; safe to drop REST.
- Order book imbalance (bid_size / (bid_size + ask_size)) was already computable from existing snapshot data — adding it as a column during collection avoids recomputing in Phase 3 analysis. Cached per-token in WS client and carried forward to price_signals for consistent time-series querying.
- `has_game_state` in `discovery_summary.json` was computed from `items[0]["data_source"] != "none"` — only checked the first item. A sport with mixed data sources across matches would be misclassified. Fix: check all items against a central registry of implemented sources.
- Pre-game 403/404 error spam from game-state APIs (NBA CDN, NHL API) was caused by polling immediately on collector start, hours before game time. Fix: three-state poller (WAITING → BACKOFF → LIVE) uses `scheduled_start` from config to delay polling, then exponential backoff until first HTTP 200. Distinct `GameNotStarted` exception separates "game not started" from real errors.
- Collector log files were ~150 MB per game because root logger at DEBUG captured aiosqlite (88%), httpcore (9%), httpx (1%), websockets (~23% in WS-only mode). Only ~0.2% was useful collector.* output. Fix: default file_level=INFO with third-party loggers silenced to WARNING; opt-in `--log-level DEBUG` restores full output for troubleshooting.
- Polymarket Sports WS broadcasts ALL sports on one connection with no subscription — matching to our game requires league abbreviation filtering + fuzzy team name matching + gameId lock-on (2 consecutive matches to confirm). No `slug` or `match_id` field exists. Score format varies by sport (tennis sets, cricket runs/wickets) — use best-effort "X-Y" parse with `None` fallback.
- Pre-collection smoke test (2026-03-25): low-activity WS shards (≤10 tokens, e.g. prop_3) trigger the 60s idle reconnect repeatedly because prop markets generate too little traffic to keep the connection alive. Reconnect logic handles it correctly — not a bug, but expect reconnect churn in logs for small shards. macOS lacks GNU `timeout`; use `kill` after `sleep` or `brew install coreutils` for `gtimeout`.
- Slug-based sport classification (`cbb-` prefix) is more reliable than keyword matching for CBB — Polymarket consistently uses `cbb-` prefixed slugs for college basketball events, preventing misclassification as NBA via the `"basketball"` keyword. Place slug checks before keyword loops so they take priority.
- Sports WS `"challenger"` league was silently dropped because `LEAGUE_MAP` for tennis only had `["atp", "wta"]`. When adding a new sport to Sports WS, always cross-check the sniff output for all league abbreviation variants — Polymarket may use sub-tour identifiers (e.g., `"challenger"` for ATP Challenger) separate from the main tour abbreviation.
- shadcn/ui v2 (2026) uses `@base-ui/react` instead of Radix — `asChild` prop no longer exists on Tooltip/Select triggers. Select `onValueChange` signature is `(value: string | null, eventDetails) => void` (nullable). Check component type signatures before wiring callbacks.
- Polymarket Sports WS (`wss://sports-api.polymarket.com/ws`) initially did **not** broadcast CBB (sniffed 2026-03-25 early afternoon during Nevada vs Auburn Sweet 16 halftime). Later the same day, `league=cbb` appeared on the feed (Auburn vs Nevada, gid=73445). Lesson: Sports WS coverage can expand mid-day — re-sniff before assuming a sport is permanently absent. Always sniff first before adding a new sport.
- Verify game dates in configs match the actual collection date before launching collectors — cricket configs were 7 days stale (March 18 games collected on March 25), producing 0 trades and 0 events.
- Cricket IPL events were misclassified as soccer because "Premier League" in "Indian Premier League" matched the soccer keyword rule before reaching the cricket rule. Fix: add `slug.startswith("cric")` prefix check (catches `cricipl-`, `cricpsl-`, `criclcl-`, `crint-`) before keyword loop, same pattern as `cbb-` prefix. When adding a new sport, always check for slug-keyword collisions with existing sport rules.
- Sports WS fuzzy team matching silently fails on diacritics — config team names from Gamma API may have Unicode characters (e.g., "Türkiye") while the WS feed sends ASCII equivalents ("Turkey"/"Turkiye"). Fix: ASCII-fold both sides with `unicodedata.normalize('NFKD')` + strip combining characters before comparing. Applies to any sport with non-ASCII team/player names (soccer, cricket, tennis).
- NBA CDN (`cdn.nba.com`) returns transient 403s from cloud IPs (Oracle Cloud ca-toronto-1). The 403 is **not permanent** — retesting minutes later with the same minimal UA + Referer succeeds. The `nba_client.py` mapped 403 to `GameNotStarted` with no warning log, causing the poller to silently back off (30s→60s→120s cap). Fix: (1) added explicit warning log on 403, (2) reduced backoff from 30-120s to 10-30s so transient blocks recover faster. Two of three NBA games recovered within ~15 min; the third was stuck in long backoff cycle.
- Sports WS fuzzy team matching fails on short college abbreviations (TX, FL, MN) because they aren't substrings of the full name and are too short for the 3-char token filter. The Sports WS sends abbreviated team names for CBB (e.g., `PUR`, `TX`, `ILL`, `HOU`). Prefix matching handles most (PUR→Purdue, HOU→Houston, ILL→Illinois) but not consonant-based abbreviations (TX→Texas). Fix: added (1) prefix matching — short name is a prefix of any word in the long name, and (2) consonant-abbreviation matching — strip vowels from each word in the long name and check if the short name matches the consonant prefix (TX→T(e)x(a)s, FL→F(l)or(i)d(a), MN→M(i)n(nesota)).
- WTA (`wta` league) is **not broadcast** on the Polymarket Sports WS as of 2026-03-26. Only ATP and Challenger tennis leagues appear. WTA matches will collect market data (snapshots, trades, price signals) but cannot get game state events via Sports WS. Need an alternative data source for WTA or wait for Polymarket to add it.
- NHL score change detection produced 0 `score_change` events across two simultaneous games (CHI-PHI, MIN-FLA) despite capturing `period_end` and `timeout` events through 3 periods. The NHL API game state poller detects period transitions but may have a bug in score diff tracking. Investigate whether the NHL client's `_last_home_score`/`_last_away_score` tracking is initialized correctly from the first poll.
- Polymarket WS has periodic global disconnects (~23:26 and ~00:23 UTC observed 2026-03-26) that hit all connections simultaneously. All shards reconnected within 1-6 seconds with 0 data gaps. The existing reconnect + backoff logic handles this seamlessly — no fix needed.
- Sports WS league coverage expands over time. New leagues observed on 2026-03-26 that weren't in earlier sniffs: `cwbb` (women's college basketball), `ufc`, `fif` (FIFA/international soccer), `lol` (League of Legends). Always re-sniff before assuming a league is absent. Add new discoveries to `LEAGUE_MAP` in `sports_ws_client.py`.
